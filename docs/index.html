1. ¿Qué es y para qué sirve?

Los demás conectores de Ruvic (Salesforce, HubSpot, GitHub...) están hechos a la medida de un proveedor específico. Este conector es distinto: es una herramienta genérica para conectarse a cualquier API REST que el cliente use y que todavía no tenga un conector dedicado.

Cada vez que se configura, queda atado a una sola API destino (definida por su URL base) — no es un conector que pueda llamar a cualquier URL que se le ocurra al agente en el momento.

Operaciones soportadas:

Operación Qué hace
http.getConsultar datos
http.postCrear datos
http.putActualizar/reemplazar datos
http.deleteEliminar datos ⚠️
http.requestVerbo arbitrario (ej. PATCH)
Este conector SÍ elimina datos A diferencia de los demás conectores de Ruvic, este sí ejecuta operaciones DELETE reales si la API destino lo soporta. No hay una capa de bloqueo — la responsabilidad de usarlo con cuidado recae en quien lo configure y en cómo se instruya al agente.

2. Cómo funciona la seguridad (léelo primero)

Porque este conector puede apuntar a cualquier API que un admin configure, trae protecciones activas contra un riesgo específico llamado SSRF (Server-Side Request Forgery): que, por error de configuración o por manipulación del agente, el conector termine llamando a direcciones internas de la infraestructura de Ruvic en vez de a la API externa que se pretendía.

Protección 1 — Atado a un solo dominio

El conector solo puede llamar al dominio configurado en URL base, más los que se agreguen explícitamente en Dominios adicionales permitidos. Si el agente intenta construir una URL hacia otro dominio, la petición se rechaza automáticamente.

Protección 2 — Bloqueo de redes privadas

Por defecto, si el dominio configurado resuelve a una dirección IP privada, interna, o al endpoint de metadata de la nube (169.254.169.254, usado por AWS/GCP/Azure para exponer credenciales de la máquina), la petición se bloquea. Esto se puede desactivar con Permitir redes privadas/internas — solo hazlo si hay una necesidad de negocio real (ej. integración con un sistema interno del cliente vía VPN).

Protección 3 — Los errores de la API destino no rompen nada

Un 404 o 500 de la API externa se devuelve como parte de la respuesta normal (no como una falla del conector), para que el agente pueda decidir cómo manejarlo — igual que lo haría un desarrollador leyendo el código de estado HTTP.

3. Prerrequisitos

  • La URL base exacta de la API destino (ej. https://api.miempresa.com).
  • Las credenciales que esa API requiera (usuario/contraseña, un token Bearer, o una API key), según lo que su documentación indique.
  • Saber si esa API necesita algún header especial siempre presente (ej. X-Tenant-Id), para ponerlo en "Headers por defecto".

4. Cómo obtener las credenciales del sistema destino

Como este conector se conecta a cualquier API que configures, no hay una consola de fabricante fija que documentar aquí — el proceso varía según el sistema. Pasos generales que aplican casi siempre:

  1. Identifica el panel de administración/desarrollador del sistema destino (suele estar en "Settings", "Developer", "API", o "Integrations").
  2. Genera una credencial dedicada para esta integración — no reutilices una contraseña personal. Según el sistema, será un usuario y contraseña, un token, o una API key.
  3. Anota la URL base de su API (a veces distinta de la URL de la app web, ej. api.sistema.com en vez de app.sistema.com).
  4. Revisa si exige headers adicionales (ej. un identificador de cuenta/tenant) que debas poner en "Headers por defecto".
  5. Guarda la credencial de forma segura — la vas a pegar en el formulario de Ruvic en la sección 5.

Si integras un sistema específico frecuentemente, considera pedir que se construya un conector dedicado en vez de usar este genérico — tendrá una guía paso a paso real de ese fabricante.

5. Permisos requeridos

Como este conector no tiene un catálogo de permisos propio (los permisos reales los define el sistema destino), sigue el principio de mínimo privilegio al generar la credencial:

  • Si el sistema permite crear usuarios/tokens con alcance limitado (ej. "solo lectura", "solo estos endpoints"), úsalo en vez de un acceso administrativo completo.
  • Otorga solo los permisos que las operaciones que vas a usar realmente necesitan — si el agente solo va a leer datos (get), no le des una credencial con permisos de escritura.
  • Si vas a habilitar delete(), confirma que entiendes exactamente qué puede borrar esa credencial en el sistema destino.

6. Configuración en Ruvic

Entra a Settings → Conectores, busca la tarjeta HTTP Genérico y pulsa Configurar. Primero elige el modo de autenticación según lo que la API destino requiera:

Modo Cuándo usarlo
Sin autenticaciónAPIs públicas sin login (ej. servicios de datos abiertos)
Basic AuthLa API pide usuario y contraseña en cada petición
Bearer TokenLa API usa un token fijo tipo Authorization: Bearer xxx
API KeyLa API pide una llave en un header o parámetro específico (ej. X-API-Key)
OAuth2 Client CredentialsLa API usa el flujo OAuth2 máquina-a-máquina (te da un token_url, client_id y client_secret)

Completa la URL base y las credenciales correspondientes. Los campos avanzados (headers por defecto, dominios adicionales, redes privadas) déjalos vacíos/por defecto salvo que sepas que los necesitas.

Guarda y pulsa Probar conexión.

7. Verificación

  1. Test de conexión exitoso.

    El botón Probar conexión debe responder: "Conexión exitosa a [tu URL] (auth_mode=...)". Nota: incluso si la API responde con un error HTTP, la conexión cuenta como exitosa — solo confirma que el host es alcanzable.

  2. Prueba mínima con el agente.

    Pide algo de solo lectura primero, por ejemplo:

    "Haz una consulta GET a /algún-endpoint de esa API y muéstrame la respuesta."

8. Solución de problemas

Mensaje / síntoma Causa probable y solución
"Destino no permitido..." El agente intentó llamar a un dominio distinto al configurado. Si es legítimo, agrégalo a "Dominios adicionales permitidos"; si no, es la protección funcionando correctamente.
"Destino bloqueado por seguridad... dirección privada/interna" El host configurado resuelve a una IP privada. Si es un uso interno intencional, activa "Permitir redes privadas/internas"; si no lo esperabas, revisa la URL base configurada.
"No se pudo conectar..." (timeout/red) Verifica que la URL base sea correcta y que el servicio destino esté disponible; puede requerir estar en una red/VPN específica.
La API responde 401/403 dentro de un resultado "exitoso" Esto es normal: el conector se conectó bien, pero las credenciales configuradas no son válidas para esa API. Revisa usuario/contraseña, token, o API key según el modo de autenticación elegido.

9. Preguntas frecuentes

¿Puedo usar el mismo conector para varias APIs distintas?

No directamente — cada configuración de este conector está atada a una URL base. Si necesitas integrarte con varias APIs distintas, configura una instancia separada del conector por cada una.

¿Por qué un 404 de la API no aparece como error en Ruvic?

Porque muchas veces un 404 es una respuesta válida y esperada (ej. "este usuario no existe"), no una falla técnica. El agente recibe el código de estado y decide cómo interpretarlo, igual que lo haría un programa normal.

¿Dónde quedan guardadas las credenciales?

Como secreto en el almacén de credenciales de la plataforma. La interfaz nunca vuelve a mostrarlas.

Buenas prácticas Usa siempre las credenciales de mínimo privilegio que la API destino permita; no actives "Permitir redes privadas/internas" sin una razón concreta; y si el agente va a poder usar delete(), dale instrucciones explícitas de siempre confirmar antes de eliminar algo.
